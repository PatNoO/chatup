# Audit → Tickets — ChatUp

Read the audit findings from the current session and create one ticket per finding in `TICKETS.md`.

**Every ticket is created by invoking the `create-ticket` skill** — not by writing to TICKETS.md directly. This guarantees every ticket follows the exact same ChatUp format, CU numbering, and templates.

---

## Step 1 — Extract findings from the audit report

Look back in the current conversation for the most recent `/audit` output. Extract every finding under these three sections:

| Audit section | Ticket type | Priority |
|---|---|---|
| 🔴 Blockers | `bug` or `chore` | high |
| 🟡 Warnings | `bug` or `chore` | medium |
| 🟢 Suggestions | `chore` or `refactor` | low |

**Type classification per finding:**
- Runtime crash risk → `bug`
- Memory leak → `bug`
- Architecture violation → `chore` or `refactor`
- Missing DI wiring → `chore`
- Callback instead of coroutine → `refactor`
- Dead code / debt → `chore`
- Security concern → `bug`

If no audit report exists in the conversation, stop and say:
> "No audit report found in this session. Run `/audit` first, then run `/audit-ticket`."

---

## Step 2 — Process findings one by one using create-ticket

For **each finding**, invoke the `create-ticket` skill with a fully formed argument string.

Do not write to `TICKETS.md` directly.
Do not batch multiple findings into one ticket.
Let `create-ticket` handle CU numbering and templates — every time.

### Argument format to pass to `create-ticket`

```
<type>: <concise title describing the problem>

Audit severity: <🔴 Blocker | 🟡 Warning | 🟢 Suggestion>
Priority: <high | medium | low>
File: <path/to/File.kt, line N if known>
Layer: <data | domain | ui | di | all>
Category: <Architecture | Security | Coroutines | DI | Memory | Dead Code | Naming>

Problem:
<Exact description of the issue from the audit report>

Why it matters:
<One sentence on the risk or impact>

Suggested fix:
<Targeted fix — not a full rewrite>

Source: Generated by /audit-ticket from codebase audit.
```

---

## Step 3 — Order of creation

Create tickets in severity order — Blockers first, then Warnings, then Suggestions. This ensures CU numbers reflect priority, with the most critical issues getting the lowest numbers in the batch.

---

## Step 4 — Report back

```
## Tickets Created — ChatUp Audit

| Priority   | CU#   | Title                              |
|------------|-------|------------------------------------|
| 🔴 High    | CU-N  | [title]                            |
| 🟡 Medium  | CU-N  | [title]                            |
| 🟢 Low     | CU-N  | [title]                            |

Total: N tickets created (X high, Y medium, Z low)
```

---

## Rules

- Always delegate to `create-ticket` — never write to TICKETS.md directly
- One ticket per finding — never bundle multiple issues
- Never create a ticket for something not explicitly in the audit report
- If a finding has no file path or description, skip it and note it was skipped
- Do not create duplicate tickets — if the same issue appears twice, create it once
